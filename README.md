# ccsearch

Hybrid semantic + keyword search over Claude Code sessions and project documentation.

Built in Rust. Zero external dependencies at runtime — uses the local Darwin embedding server (Qwen3-Embedding-4B on GTX 1650) for semantic search, plain substring matching for keyword search.

## Architecture

```
ccsearch index   →  Scans sessions-index.json + docs/**/*.md
                    Embeds text via Darwin LAN (192.168.4.1:8080)
                    Writes binary index (~1-5 MB)

ccsearch "query" →  Loads index (<1ms)
                    Embeds query via Darwin (~70ms)
                    Cosine similarity + substring match (~1ms)
                    Prints top-10 results
                    Total: ~75ms end-to-end
```

### Why not SQLite?

118 sessions + ~656 docs = ~774 entries. At 1024 dims × 4 bytes × 774 = **3 MB** of vectors. Fits trivially in memory. SQLite adds schema management, SQL parsing overhead, and a C compilation dependency — all for data that fits in a single `Vec<Entry>`.

### Why not Tantivy?

Tantivy (what Emanuel's CASS uses) is a Rust full-text search engine optimized for millions of documents. At our scale (~1000 entries), simple substring matching is instant. Adding Tantivy would increase binary size from 2.4 MB to ~15 MB and add build complexity for zero perceptible benefit.

### Why Rust?

This is a CLI tool that should feel instant. Python takes 50-100ms just to start the interpreter. Rust starts in <5ms. The computation (cosine similarity over ~1000 vectors) is pure CPU math.

### Advantage over CASS

Emanuel's CASS (Coding Agent Session Search) uses Tantivy for BM25 full-text search only. ccsearch adds **semantic search** via the local embedding model — "find the session about USB driver debugging" matches sessions mentioning "RTL8153 DWC3 DMA underrun" even without exact keyword overlap. Hybrid search (keyword + semantic) is strictly more capable than keyword-only.

## Usage

```bash
# Build the index (scans sessions + docs, embeds new entries)
ccsearch index

# Search everything (sessions + docs, hybrid semantic + keyword)
ccsearch "chezmoi drift detection"

# Sessions only
ccsearch search -s "temporal workflow debugging"

# Docs only
ccsearch search -d "network topology"

# Keyword-only (no embedding call, instant)
ccsearch search -k "rtl8153"

# Show index stats
ccsearch stats
```

## What gets indexed

### Sessions
All CC session summaries from `~/.claude/projects/*/sessions-index.json`. Each session has a summary (generated by CC) and a first prompt. Both are embedded together.

### Docs
All `.md` files in:
- `~/Projects/*/docs/**/*.md` — project documentation
- `~/dotfiles/docs/**/*.md` — dotfiles/infrastructure docs

Excluded: files >100 KB, hidden dirs, archive/vendor/node_modules.

First 800 characters of each doc are embedded (stays within the 512-token context of the embedding server).

## Configuration

| Env var | Default | Description |
|---------|---------|-------------|
| `CCSEARCH_EMBEDDING_URL` | `http://192.168.4.1:8080/v1/embeddings` | Embedding API endpoint |

## Index storage

Binary JSON file at `~/.local/share/ccsearch/index.bin`. Contains all metadata + embedding vectors. Typically 1-5 MB.

Incremental: re-running `ccsearch index` only embeds new entries (cached by ID). Use `--force` to reindex everything.

## Build

```bash
cd ~/Projects/ccsearch
cargo build --release
cp target/release/ccsearch ~/.local/bin/
```

## Dependencies

Build-time only:
- `clap` — CLI argument parsing
- `serde` + `serde_json` — serialization
- `ureq` — HTTP client (sync, no async runtime)
- `glob` — file pattern matching

Runtime: none (the embedding server must be reachable on the network).
